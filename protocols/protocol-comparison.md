# 主流代理协议横向对比（2026）

> **摘要**: 本文对比当前主流的代理协议——Shadowsocks、VMess、VLESS+Reality、Trojan、Hysteria2、NaiveProxy、VLESS+XHTTP——从加密方式、抗检测能力、性能表现、部署难度四个维度进行系统评估，帮助用户根据自身需求选择合适的协议。无论你是刚入门的新手还是有经验的运维人员，本文都将提供有价值的参考。

## 为什么需要这篇对比

市面上的协议对比文章大多停留在"推荐用 XX"的结论层面，缺乏对底层原理的解释。不同协议在不同网络环境下表现差异很大，理解它们的设计思路比记住结论更重要。

GFW 的检测能力在持续进化，2024 年以来对裸 VMess 的精准封锁、对 QUIC 流量的深度分析都表明：单纯依赖某个"最强协议"的思路已经过时。真正有效的策略是理解每个协议的优势边界，根据实际网络环境灵活组合。

本文不会告诉你"用这个就行了"，而是帮你建立判断框架——当环境变化时，你能自己做出正确选择。

## 逐一解析

### Shadowsocks (2022 Edition / AEAD-2022)

**设计思路**：

Shadowsocks 的核心设计理念是"轻量级加密代理"——不做伪装，只做加密转发。它不试图让流量看起来像正常的 HTTPS 或其他协议，而是通过加密让审查者无法解析流量内容。

早期版本（stream cipher、AEAD）存在严重的安全缺陷：重放攻击可以让审查者录制一段加密流量后重新发送给服务器，通过观察服务器的响应行为判断是否为 Shadowsocks 服务；主动探测则允许审查者直接向疑似 SS 的端口发送特定构造的数据包，根据服务器的错误响应来确认协议类型。

AEAD-2022（即 Shadowsocks 2022 Edition）对这些问题做了根本性修复：引入了基于时间戳的重放过滤机制，每个数据包都绑定了时间窗口，过期数据包直接丢弃；对于无法解密的请求，服务器不再返回任何有辨识度的错误响应，而是直接静默关闭连接，使主动探测失去依据。

**优势**：

- 性能开销极低，加解密效率高，延迟表现在所有协议中属于第一梯队
- 客户端生态极其成熟，几乎所有主流代理客户端都支持（Clash、Surge、Shadowrocket、sing-box 等）
- AEAD-2022 修复了旧版的重放攻击和探测漏洞，安全性达到了可接受的水平
- 协议设计简洁，服务端配置简单，运维成本低

**劣势**：

- 没有任何 TLS 伪装层，流量不像 HTTPS 或其他常见协议
- 虽然审查者无法解密内容，但流量的统计特征（包长分布、连接时序模式）仍然可以被机器学习模型识别
- 在某些 ISP 或敏感时期，高熵加密流量本身就可能触发封锁策略

**适用场景**：

对延迟敏感的使用场景（如游戏加速、实时通信），以及网络环境相对宽松、审查力度不大的地区。如果你的线路质量好、ISP 对加密流量不做针对性干扰，SS 2022 仍然是性价比最高的选择。

### VMess

**设计思路**：

VMess 是 V2Ray 项目的原生协议，设计目标是提供自带加密和身份认证的代理通信。它使用 UUID 作为用户标识，通过自定义的加密方案实现数据传输。VMess 在设计时考虑了多种加密方式（AES-128-GCM、ChaCha20-Poly1305 等），并在协议层面内置了时间戳校验来防止重放攻击。

**优势**：

- 客户端生态最为完善，几乎所有代理客户端都将 VMess 作为第一个支持的协议
- 配置简单直观，对新手友好
- V2Ray/Xray 社区庞大，遇到问题容易找到解决方案

**劣势**：

VMess 的问题在当前环境下已经非常突出：

1. **安全设计缺陷**：VMess 的加密方案是自行设计的，未经过严格的密码学审计。协议头部的结构存在可识别特征，研究人员已经公开了针对 VMess 的流量识别方法。

2. **GFW 精准识别**：未套 TLS 的裸 VMess 流量已经可以被 GFW 精准识别和封锁。即便是加密后的 VMess 数据包，其长度分布和握手行为都有明显的协议指纹。

3. **TLS 指纹问题**：即使套上 TLS，VMess 的客户端在 TLS 握手时产生的 JA3 指纹与真实浏览器存在差异。GFW 可以通过 JA3/JA4 指纹分析来识别代理流量，导致 TLS 层的伪装效果大打折扣。

**为什么不再推荐**：

VMess 在设计之初（2017 年前后）是一个合理的选择，但以 2026 年的标准来看，它在安全性和抗检测能力上都已经落后。GFW 对 VMess 的识别准确率很高，即使在低审查时期也可能被标记。更重要的是，VLESS+Reality 在各方面都是 VMess 的上位替代——更安全、更难检测、性能更好。如果你还在使用 VMess，强烈建议尽快迁移。

### VLESS + Reality

**设计思路**：

VLESS+Reality 是当前最受推荐的协议组合。VLESS 本身是一个极简的代理协议——去掉了 VMess 中多余的加密层（因为 TLS 已经提供了加密），只保留最基本的认证和数据传输功能。而 Reality 则是 TLS 伪装层面的一次根本性创新。

传统的 TLS 伪装（如 Trojan）需要你拥有一个真实的域名和对应的 TLS 证书。Reality 的核心思路是：**借用目标网站的 TLS 证书来完成伪装，而不需要你自己申请任何证书或拥有任何域名**。

**Reality 如何工作**：

整个过程可以这样理解：

1. **客户端发起连接**：客户端向你的代理服务器发起 TLS 握手，但 SNI（Server Name Indication）填写的是目标伪装站点的域名，比如 `apple.com`。

2. **服务端响应**：代理服务器收到握手请求后，会去目标站点获取其真实的 TLS 证书，然后用这个真实证书完成 Server Hello 响应。从外部观察者（包括 GFW）的角度来看，这个 TLS 握手与直接访问 `apple.com` 完全一致——因为证书确实是真的。

3. **身份验证**：在 TLS 握手完成后，客户端和服务端通过预共享的 shortId 和私钥（基于 x25519 曲线）完成真正的身份认证。只有持有正确密钥的客户端才能继续通信，其他任何探测请求都会被服务器当作普通的 HTTPS 请求处理（返回目标站点的真实网页内容）。

4. **数据传输**：认证完成后，VLESS 协议在已建立的 TLS 隧道内传输代理数据。

这意味着 GFW 看到的始终是一个到 `apple.com` 的正常 TLS 连接，使用的是真实的证书，TLS 指纹也与 Xray 内置的 uTLS 库模拟的浏览器指纹一致。

**为什么常选 apple.com 做 dest 目标站**：

选择 dest 目标站有几个原则：流量体量足够大（这样你的连接混在海量真实流量中不显眼）、目标域名在审查国家不太可能被封锁（苹果的服务在中国大陆、俄罗斯、伊朗都正常运营）、以及目标站本身支持 TLS 1.3。`apple.com`、`microsoft.com`、`www.samsung.com` 等大型科技公司域名都是常见选择。

**优势**：

- 抗检测能力目前最强之一，TLS 握手与真实访问目标站点不可区分
- 无需购买域名、无需申请和维护 TLS 证书，大幅降低部署和运维成本
- VLESS 协议本身极轻量，性能开销低
- uTLS 库可以模拟 Chrome、Firefox 等主流浏览器的 TLS 指纹，解决了 JA3 识别问题
- 对主动探测具有天然免疫力——未认证的连接会收到目标站点的真实网页

**劣势**：

- 部分老旧客户端或移动端客户端对 Reality 的支持不够完善
- 配置参数较多（dest、serverName、shortId、privateKey/publicKey），对新手有一定门槛
- dest 目标站的选择需要一定经验，选错可能导致连接异常或被针对性检测

**适用场景**：

当前大多数用户的最佳选择。无论是个人使用还是为他人提供服务，VLESS+Reality 都提供了最好的安全性与易用性平衡。除非你有特殊的带宽需求（考虑 Hysteria2）或极端的隐蔽性需求（考虑 VLESS+XHTTP+Reality），否则 VLESS+Reality 就是你的默认选项。

### Trojan

**设计思路**：

Trojan 的设计目标是让代理流量完全模仿正常的 HTTPS 流量。它直接工作在 TLS 层之上，使用真实的 TLS 证书（通过 Let's Encrypt 等 CA 签发），代理数据通过标准的 TLS 加密通道传输。从网络层面观察，Trojan 的流量与访问一个 HTTPS 网站几乎没有区别。

**优势**：

- 流量特征与正常 HTTPS 高度相似，对普通的流量审查有很好的隐蔽性
- 协议设计简洁，没有多余的加密层（TLS 本身就提供了加密）
- 未认证的连接会被转发到一个真实的网站，提供了一定的主动探测防御能力
- 客户端支持广泛

**劣势**：

- 需要拥有域名并申请、维护 TLS 证书（Let's Encrypt 证书每 90 天需要续期）
- TLS 指纹仍然可被分析——如果客户端使用的 TLS 库与真实浏览器有差异，JA3/JA4 指纹会暴露代理特征
- 主动探测风险依然存在：虽然未认证连接会被转发，但审查者可以通过分析转发行为的细微差异（响应时延、HTTP 头顺序等）来判断是否为 Trojan 服务器
- 域名本身可能被 DNS 污染或 SNI 封锁，一旦域名暴露，整个节点失效

### Hysteria2

**设计思路**：

Hysteria2 是一个基于 QUIC（HTTP/3 的底层传输协议）构建的高性能代理协议。与前面所有基于 TCP 的协议不同，Hysteria2 走的是 UDP 路线。它利用 QUIC 的多路复用和 0-RTT 握手特性来实现低延迟连接，同时内置了一套自定义的拥塞控制算法（Brutal），可以在丢包率较高的线路上维持较高的传输速率。

简单来说，Hysteria2 的思路是：既然 QUIC/HTTP3 已经是互联网的趋势，那代理流量也可以伪装成 QUIC 流量。

**优势**：

- 在高带宽场景下表现极其出色，内置的 Brutal 拥塞控制可以在丢包环境中保持高吞吐
- 连接建立速度快（QUIC 0-RTT），适合频繁建立新连接的场景
- 多路复用能力强，单连接即可承载多个并发请求
- 部署相对简单，配置参数少

**劣势**：

- UDP 流量在部分运营商网络中会被 QoS 限速，尤其是在跨境链路上。某些 ISP 甚至会直接封锁所有 UDP 流量
- QUIC 流量的特征正在被 GFW 研究。虽然目前尚未实现精准封锁，但 QUIC 的握手过程有一些固定特征（Initial Packet 的格式、版本协商等），未来被针对的风险不可忽视
- 在 UDP 被严格限制的网络环境下（如部分企业网络、学校网络），Hysteria2 可能完全无法使用

**适用场景**：

网络环境允许 UDP 且对带宽有高要求的场景——流媒体、大文件下载、视频通话等。如果你的 ISP 对 UDP 友好且线路质量一般（丢包较高），Hysteria2 的 Brutal 拥塞控制能带来明显优于 TCP 协议的体验。但一定要准备 TCP 协议作为后备方案。

### NaiveProxy

**设计思路**：

NaiveProxy 采用了一个极为大胆的方案：直接嵌入 Chromium 的网络栈来处理代理流量。这意味着从 TLS 握手到 HTTP/2 连接管理，NaiveProxy 产生的每一个网络数据包都与真正的 Chrome 浏览器发出的完全一致——因为它用的就是 Chrome 的代码。

这是一个从根源上解决 TLS 指纹问题的思路：你不需要模拟浏览器的行为，因为你就是浏览器。

**优势**：

- 理论上具有最强的抗检测能力。TLS 指纹、HTTP/2 帧结构、连接行为都与真实 Chrome 浏览器完全一致
- 对 JA3/JA4 指纹分析完全免疫
- 流量不仅加密内容不可解析，连元数据特征都与正常浏览器流量无法区分

**劣势**：

- 编译和维护成本极高。NaiveProxy 需要从 Chromium 源码编译，而 Chromium 的代码库非常庞大（完整拉取超过 30GB），编译时间长、资源消耗大
- Chromium 更新频繁，每次上游更新都可能引入 API 变更或兼容性问题，导致 NaiveProxy 需要跟进适配
- 客户端选择极其有限，不像 VLESS 或 SS 那样有丰富的第三方客户端支持
- 服务端也需要特殊的 Caddy 分支（forwardproxy 插件），部署流程不够标准化
- 对于大部分用户而言，NaiveProxy 的抗检测优势相对于 VLESS+Reality 并不明显，但维护成本却高出很多

---

### VLESS + XHTTP + Reality + XMUX

**设计思路**：

这是目前代理协议技术栈的"天花板"组合。它在 VLESS+Reality 的基础上更进一步：通过 XHTTP 传输层将代理数据伪装成普通的 HTTP 请求（而不仅仅是 TLS 加密隧道），并通过 XMUX 实现连接的多路复用。

传统的 TLS 隧道方案（包括 VLESS+Reality）在行为层面仍然有一个特征：客户端与服务器之间会建立长时间的 TLS 连接并持续传输数据，这种"长连接"模式与普通的网页浏览行为有差异。XHTTP 的做法是将代理流量拆分为一系列短的 HTTP 请求和响应，模仿正常的网页浏览模式——就像用户在浏览网站、加载资源一样。

XMUX 则解决了多路复用的问题，允许多个代理连接共享同一组底层 HTTP 连接，提高了资源利用效率。

**优势**：

- 抗检测能力在当前所有方案中最强，不仅 TLS 层面不可区分，连流量行为模式都与真实浏览高度相似
- 多路复用支持减少了连接数量，降低了被统计分析识别的风险
- 结合 Reality 的证书借用机制，无需域名和证书

**劣势**：

- 部署复杂度是所有方案中最高的，需要同时配置 VLESS、XHTTP、Reality、XMUX 多个组件
- 客户端支持有限，目前主要依赖 Xray-core 的最新版本
- 技术栈较新，实际生产环境中的稳定性和边界情况还在持续检验中
- 对运维人员的技术水平要求较高，排错难度大

## 如何选择

```
你的网络环境是否封锁 UDP？
├── 是 → 排除 Hysteria2
├── 否 → Hysteria2 可作为备选
│
你是否在高审查地区（中国大陆/伊朗/俄罗斯）？
├── 是 → 推荐 VLESS+Reality 或 VLESS+XHTTP+Reality
├── 否 → Shadowsocks 2022 / Trojan 通常足够
│
你是否需要极简部署？
├── 是 → VLESS+Reality（无需域名证书）
├── 否 → 根据抗检测需求选择
```

**补充建议**：

- **入门用户**：直接选 VLESS+Reality。教程多，部署不需要域名和证书，抗检测能力足够应对绝大多数场景。
- **追求带宽**：在主力节点使用 VLESS+Reality 的同时，准备 Hysteria2 作为 UDP 友好环境下的加速选项。
- **极端隐蔽需求**：VLESS+XHTTP+Reality+XMUX，但请确保你有足够的技术能力来维护。
- **始终准备备用方案**：不要只依赖一种协议。敏感时期多协议切换是最基本的运维策略。

## 常见问题

### Q: VMess 还能用吗？

可以用，但不推荐。VMess 在技术上仍然可以工作，大部分客户端也仍然支持它。但问题在于：GFW 已经具备了对裸 VMess 流量的精准识别能力，即使套上 TLS，JA3 指纹问题也使得伪装效果不理想。更重要的是，VMess 存在已知的安全设计缺陷，且上游项目不太可能再进行根本性的修复。VLESS+Reality 在几乎所有维度上都是 VMess 的上位替代，迁移成本也不高——如果你在用 Xray，只需要修改配置文件即可。

### Q: Reality 的 dest 目标站如果被封了怎么办？

首先，选择 dest 目标站时就应该优先选择大型、广泛使用且不太可能被封锁的站点——比如苹果、微软、三星等国际科技公司的域名，这些站点在中国大陆有大量正常业务流量，封锁它们的代价对审查方来说也很高。

但即便如此，也要做好准备：

1. **提前准备多个 dest 配置**，至少 3-5 个备选目标站，确保可以快速切换
2. **定期监控 dest 站点的可达性**，不要等到用户反馈才发现问题
3. **选择 dest 时验证目标站是否支持 TLS 1.3 和 H2**，这是 Reality 正常工作的前提
4. **避免使用冷门站点**——流量太少的目标站反而容易因流量异常被注意到

### Q: 协议选择和节点质量哪个更重要？

节点质量和运营敏捷性比协议选择更重要。一个运行在优质线路上、由经验丰富的运维人员管理的 Shadowsocks 节点，实际使用体验几乎一定好于一个部署在劣质 VPS 上、无人维护的 VLESS+Reality 节点。

协议只是安全架构的一部分，而且往往不是最薄弱的那部分。真正决定节点存活时间和使用体验的因素包括：IP 质量（是否已被标记）、线路稳定性、运维响应速度（封锁后多快能切换）、以及是否有完善的监控和自动化切换机制。

所以正确的优先级应该是：**运营敏捷性 > 线路质量 > 协议选择**。先确保你有快速响应封锁的能力，再考虑协议层面的优化。

### Q: Hysteria2 和 VLESS+Reality 哪个快？

取决于具体场景，不能一概而论：

- **UDP 友好 + 高丢包线路**：Hysteria2 明显更快。它的 Brutal 拥塞控制算法专门为高丢包环境设计，可以维持远高于 TCP 的吞吐量。对于流媒体、大文件下载等高带宽场景，Hysteria2 的优势很明显。

- **UDP 受限或正常线路**：VLESS+Reality 更稳定。在 UDP 被 QoS 限速的网络中，Hysteria2 的性能可能反而不如 TCP 方案。VLESS+Reality 基于 TCP，在绝大多数网络环境中都能正常工作，不受 UDP 限制的影响。

- **延迟敏感场景**：两者差距不大。QUIC 的 0-RTT 握手在理论上有优势，但实际使用中这个差距很难感知到。

**实用建议**：两者并不互斥，可以在同一台服务器上同时部署。日常使用 VLESS+Reality 保证稳定性，需要高带宽时切换到 Hysteria2。
