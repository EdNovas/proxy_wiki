# 从 VMess 到 VLESS：协议演进史

> **摘要**: 从 2015 年的 VMess 到 2023 年的 VLESS+Reality+XHTTP，代理协议经历了快速演进。每一次升级都是为了应对 GFW 不断增强的检测能力。本文梳理这段演进历史，帮助你理解为什么当前推荐 VLESS+Reality。

## 为什么要了解协议演进

很多人选择代理协议的逻辑是"别人推荐什么就用什么"。这在大部分时间没问题，但当封锁来临、节点批量失效时，不理解协议原理的用户往往无法判断问题出在哪里，也无法做出有效的应对决策。

理解协议演进的过程，本质上是理解 GFW 检测手段的进化过程。每一次协议升级都不是凭空发生的——它背后对应着一种新的封锁策略。掌握了这条时间线，你就能建立起"为什么要用这个协议"的判断框架，而不只是记住一个结论。

本文按时间线梳理从 VMess 到 VLESS+Reality 的完整演进历程，覆盖关键技术节点和社区事件。

---

## 2015-2017: VMess 时代

2015 年，Victoria Raymond 发布了 V2Ray 项目，其核心协议 VMess（V2Ray Message Protocol）成为 Shadowsocks 之后最重要的代理协议。

VMess 的设计思路与 Shadowsocks 有根本区别。SS 追求极简——只做加密转发；而 VMess 在协议层面加入了更多功能：基于 UUID 的用户认证、自带的加密层（支持 AES-128-GCM、ChaCha20-Poly1305 等多种加密方式）、以及指令头的时间戳校验机制来防止重放攻击。

在当时的环境下，VMess 解决了几个关键问题：

- **多用户管理**：通过 UUID 区分用户，不需要像 SS 那样为每个用户分配独立端口
- **协议层安全性**：自带认证和加密，减少了对外层传输协议的依赖
- **灵活的传输层**：支持 TCP、mKCP、WebSocket 等多种传输方式，可以根据网络环境灵活选择

V2Ray 还引入了路由规则的概念——虽然功能相对基础，但为后来 Clash 的规则分流体系奠定了思路基础。

这一阶段的 GFW 主要依靠端口封锁和 IP 黑名单，对协议层面的识别能力有限。VMess 的设计在当时是足够安全的。

---

## 2018-2019: Clash 与生态成熟

2018 年，Dreamacro 发布了 Clash，这个项目带来的变革不在协议层面，而在用户体验层面——它将代理从"能用"推向了"好用"。

Clash 的核心创新是基于规则的流量分流：

- 根据域名、IP、地理位置、进程名等条件，将不同的流量路由到不同的代理节点或直连
- 支持代理组（Proxy Group）概念，实现负载均衡、故障切换、延迟测速等高级功能
- YAML 配置文件格式让规则编写和分享变得容易

在 Clash 出现之前，大多数用户要么全局代理，要么手动切换。Clash 让"国内流量直连、国际流量走代理、特定网站走特定节点"这种精细化的分流成为普通用户也能实现的操作。

这一时期，VMess + WebSocket + TLS 成为主流部署方案。通过将 VMess 流量封装在 WebSocket 中，再用 TLS 加密，从外部看就是普通的 HTTPS 流量。配合 Nginx 反向代理，可以在同一台服务器上同时承载正常网站和代理服务。

Clash 的规则分流 + VMess/WS/TLS 的传输方案 + CDN 中转的部署架构，构成了这个时代最典型的代理方案。

但问题也在积累：VMess 协议本身的加密层与 TLS 形成了双重加密，带来了不必要的性能损耗。更重要的是，GFW 的技术也在同步进步。

---

## 2020: VLESS 与 Xray 分裂

2020 年是代理协议发展的关键转折点，技术和社区两条线同时发生了重大变化。

**技术线：XTLS 与 VLESS 的诞生**

开发者 RPRX 提出了 XTLS（X Transport Layer Security）的概念。他的核心观察是：当 VMess 运行在 TLS 之上时，内层协议的加密是多余的——TLS 已经提供了完整的加密和认证，VMess 自带的加密层只是浪费 CPU。

基于这个思路，VLESS 协议被设计出来。与 VMess 相比，VLESS 做了大幅简化：

- 去掉了协议层的加密，将加密完全交给外层的 TLS
- 精简了协议头，减少了协议指纹
- 保留了 UUID 认证机制

XTLS 则更进一步：当检测到内层流量本身已经是 TLS 加密时（比如你通过代理访问一个 HTTPS 网站），直接将内层的 TLS 数据"透传"而不是再加密一次。这从根本上消除了双重加密的性能损耗。

**社区线：V2Fly 与 Xray 的分裂**

Victoria Raymond 在 2019 年逐步淡出 V2Ray 项目后，社区维护者成立了 V2Fly 组织继续开发 v2ray-core。RPRX 将 XTLS 和 VLESS 提交给 V2Fly，但双方在技术路线和许可证问题上产生了严重分歧。

最终，RPRX 从 v2ray-core fork 出了 Xray-core，并以此为基础独立发展。Xray 成为 VLESS 和 XTLS 的主要载体，并逐渐在功能和性能上超越了原版 V2Ray。

这次分裂虽然带来了社区的割裂，但客观上加速了技术创新——两个项目的竞争推动了代理协议的快速演进。

## 2021-2022: TLS 指纹成为焦点

如果说 2020 年之前 GFW 主要靠协议特征和流量统计来识别代理流量，那么 2021 年开始，TLS 指纹检测成为了新的主战场。

**JA3/JA4 指纹**

TLS 握手过程中，客户端发送的 ClientHello 消息包含了大量信息：支持的密码套件、扩展列表、椭圆曲线参数等。JA3 算法将这些参数哈希成一个指纹值，可以用来识别客户端类型。

问题在于：Go 语言标准库的 TLS 实现产生的 JA3 指纹与主流浏览器（Chrome、Firefox）完全不同。GFW 可以通过这个指纹直接判断出"这不是浏览器发起的连接"，进而将其标记为可疑流量。

这意味着：即使你用了完美的 TLS 加密，让流量内容无法被解读，但握手阶段的指纹就已经暴露了你不是真正的浏览器。

**uTLS 的应对**

uTLS 库的出现是对这一检测手段的直接回应。它允许 Go 程序在 TLS 握手时模拟各种浏览器的 ClientHello 指纹（Chrome、Firefox、Safari 等），使代理客户端的 TLS 握手看起来与真实浏览器一致。

Xray-core 和后来的 sing-box 都集成了 uTLS，用户只需要在配置中指定 `fingerprint: chrome` 就能模拟 Chrome 的 TLS 指纹。

**Trojan 的兴与衰**

Trojan 协议在这个时期经历了从崛起到逐渐边缘化的过程。它的设计思路很直接：使用真实的 TLS 证书，在 TLS 层之上用密码认证，对外表现为一个完全正常的 HTTPS 服务。非认证请求会被转发到一个真实网站，实现完美的回落。

Trojan 的问题不在协议本身，而在部署模型：它依赖真实的域名和 TLS 证书，这带来了域名可能被封锁、证书续签可能暴露身份等运维风险。同时，Trojan 没有解决 TLS 指纹问题——如果 TLS 握手的指纹暴露了非浏览器客户端，再完美的协议设计也无济于事。

## 2023: Reality 革命

2023 年初，RPRX 发布了 Reality，这是 XTLS Vision 的进一步演进，也是代理协议发展史上的一个里程碑事件。

Reality 解决的不是单一问题，而是同时攻克了三个长期困扰代理部署的核心难题：

**第一：消除域名依赖**

传统的 TLS 伪装方案（Trojan、VLESS+TLS）都需要一个真实域名和对应的 TLS 证书。域名可能被封锁，证书申请和续签过程可能暴露服务器身份。

Reality 不需要自己的域名和证书。它通过"借用"目标网站（如 `www.microsoft.com`、`www.apple.com` 等大型网站）的 TLS 证书来完成握手。从网络流量的角度看，这个连接就是在访问微软或苹果的网站。

**第二：解决 TLS 指纹问题**

Reality 在服务端模拟目标网站的 TLS 行为，包括 ServerHello 的参数选择、证书链的呈现方式等。配合客户端的 uTLS 模拟，整个 TLS 握手过程从指纹层面看与真实访问目标网站几乎无法区分。

**第三：抵抗主动探测**

这是 Reality 最精妙的设计。当 GFW 或其他探测器主动连接 Reality 服务器时，由于探测器没有正确的认证信息（shortId 和 privateKey），Reality 服务器会将连接直接转发到目标网站，返回目标网站的真实响应。

探测者看到的完全是一个正常的网站服务器——它无法区分这到底是目标网站本身，还是一个 Reality 代理服务器。

**部署模型的优势**

Reality 的部署比传统 TLS 方案更简单：

- 不需要购买域名
- 不需要申请和维护 TLS 证书
- 不需要运行 Web 服务器做回落
- 只需要一台 VPS 和 Xray-core

这让个人自建代理的门槛大幅降低，同时安全性反而提高了。Reality 发布后迅速成为社区推荐的首选方案。

## 2024-2025: XHTTP 与 AnyTLS

Reality 解决了身份伪装的问题，但代理协议的演进并没有停止。2024-2025 年，社区在传输层面继续探索新的可能性。

**XHTTP**

XHTTP 是 Xray-core 引入的新传输方式，核心思路是将代理数据封装为标准的 HTTP 请求。与 WebSocket 不同，XHTTP 使用的是普通的 HTTP POST/GET 请求，流量模式更接近正常的网页浏览行为。

XHTTP 的优势在于穿透性：许多网络环境对 WebSocket 长连接有限制或监控，但很少会限制普通的 HTTP 请求。XHTTP 配合 Reality 使用时，流量表现为向某个大型网站发起的 HTTP 请求序列，极难被识别。

在 CDN 中转场景中，XHTTP 也比 WebSocket 表现更好——它不需要维持长连接，对 CDN 更友好，中转成本更低。

**AnyTLS**

AnyTLS 是由 anytls 项目提出的另一种思路。它试图解决的问题是：即使 TLS 握手完美伪装，长时间的加密数据传输本身的流量模式（数据包大小分布、传输间隔等）仍然可能被统计分析识别。

AnyTLS 通过在 TLS 连接内部引入填充和分片机制来扰乱流量模式，使统计分析更难以识别代理流量。这是一种传输层面的补充方案，可以与现有的协议组合使用。

这一时期还出现了 sing-box 对多种协议的统一整合。sing-box 同时支持 VLESS、VMess、Trojan、Shadowsocks、Hysteria2 等协议，提供了统一的配置格式和客户端实现，逐步成为跨平台的主流核心。

## 2026: 当前格局

截至 2026 年中，代理协议的格局已经相对稳定，形成了明确的主次梯队：

**第一梯队：VLESS + Reality**

当前社区的首选推荐。它在抗检测能力、部署便捷性、性能表现三个维度上都达到了最佳平衡。对于绝大多数用户和使用场景，VLESS+Reality 就是最优解。

配合 XTLS Vision 流控，在转发 TLS 流量时几乎没有额外的加密开销，延迟和吞吐量都非常优秀。

**第二梯队：Hysteria2（UDP 场景补充）**

Hysteria2 基于 QUIC 协议，在 UDP 友好的网络环境中性能极为出色。特别是在高丢包线路上，其 Brutal 拥塞控制算法可以维持远高于 TCP 方案的吞吐量。

Hysteria2 不是 VLESS+Reality 的替代品，而是补充。推荐的做法是在同一台服务器上同时部署两者：日常使用 VLESS+Reality，需要高带宽或 UDP 传输（如视频通话、游戏）时切换到 Hysteria2。

**统一平台：sing-box**

sing-box 正在成为跨平台的统一代理核心。它整合了主流协议支持，提供统一的配置格式，在 Android、iOS、Windows、macOS、Linux 上都有可用的客户端。相比过去不同平台使用不同核心（Windows 用 V2Ray、iOS 用 Shadowrocket）的碎片化局面，sing-box 正在推动生态的统一。

**机场生态**

商业机场普遍已经迁移到 VLESS+Reality 或 Hysteria2，少数仍在使用 VMess+WS+TLS 的机场主要是因为 CDN 中转的需求。随着 XHTTP 的成熟，这部分需求也在逐步迁移。

## 演进的规律

回顾这段历史，可以清晰地看到代理协议演进的底层逻辑：

**第一阶段：协议加密（2015-2019）**

核心策略是加密流量内容，让审查者无法解读。VMess 和 Shadowsocks 都属于这一阶段——通过加密让流量变成不可读的密文。

这一阶段的局限性在于：加密流量本身就是一个特征。当网络中出现大量高熵、无规律的加密流量时，即使审查者不知道内容是什么，也能通过统计特征将其识别为"异常流量"。

**第二阶段：流量伪装（2019-2022）**

核心策略是让代理流量看起来像正常的 HTTPS 流量。VMess+WS+TLS、Trojan 都属于这一阶段——把流量封装在 TLS 中，从外部看就是普通的 HTTPS 连接。

这一阶段的突破口被 GFW 找到了：TLS 指纹。虽然流量内容被 TLS 保护，但 TLS 握手过程本身暴露了客户端不是真正的浏览器。

**第三阶段：身份借用（2023-至今）**

核心策略是不只让流量看起来像 HTTPS，更要让整个连接看起来就是在访问一个真实的网站。Reality 代表了这一阶段的核心思路——借用真实网站的身份，让审查者即使进行主动探测也无法区分代理服务器和真实网站。

**检测与反检测的螺旋**

每一次 GFW 升级检测手段，社区就会发展出新的对抗方案；每一次对抗方案成熟，GFW 又会研究新的检测方法。这个螺旋不会停止。

值得注意的是，演进的方向始终是从"隐藏内容"到"隐藏身份"——从"让你看不懂我在做什么"变成"让你根本不知道我在这里"。

## 常见问题

### Q: VMess 彻底淘汰了吗？

技术上已经过时，但仍有大量存量用户和节点在使用。VMess 的问题不是"不能用"，而是"没有理由继续用"——VLESS 在各方面都优于 VMess，部署难度也没有增加。

不推荐新部署使用 VMess。如果你现有的节点还在用 VMess，建议在下次维护时迁移到 VLESS+Reality。迁移成本很低，主要是更新服务端和客户端配置。

### Q: 下一个革命性协议会是什么？

很难预测，但有几个方向值得关注：

- **ECH（Encrypted Client Hello）的普及**：ECH 如果被主流浏览器和服务器广泛支持，将从根本上改变 TLS 指纹检测的可行性，因为 ClientHello 中的敏感信息将被加密。
- **QUIC 协议的成熟**：随着 HTTP/3 的普及，QUIC 流量在互联网中的占比持续增长，基于 QUIC 的代理协议的"混入"效果会越来越好。
- **AI 驱动的流量分析**：GFW 如果大规模部署基于机器学习的流量分类器，现有的伪装方案可能需要根本性调整。

### Q: 我需要关心协议演进吗？

取决于你的使用方式：

- **使用商业机场**：不太需要。靠谱的机场运营者会主动跟进协议升级，你只需要保持客户端更新即可。
- **个人自建**：建议关注。至少订阅 Xray-core 和 sing-box 的 Release 页面，了解重要更新。当出现新的封锁事件时，社区通常会很快给出应对方案。
- **技术爱好者**：强烈建议深入了解。理解协议原理不仅能帮你更好地配置和排障，也能让你在封锁事件中更快做出正确判断。
