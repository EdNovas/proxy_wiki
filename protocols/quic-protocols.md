# Hysteria2 与 TUIC：基于 QUIC 的协议

> **摘要**: Hysteria2 和 TUIC 是基于 QUIC（HTTP/3 底层协议）构建的代理协议。它们利用 UDP 传输实现高带宽和低延迟，特别适合流媒体和大文件下载场景。本文解析两者的技术原理、各自特点和适用场景。

## 为什么用 QUIC

在讨论 Hysteria2 和 TUIC 之前，首先需要理解它们为什么选择了 QUIC 而不是沿用传统的 TCP。这个选择不是赶时髦，而是为了解决 TCP 在代理场景中的根本性缺陷。

### TCP 的局限

几乎所有主流代理协议——Shadowsocks、VMess、VLESS、Trojan——都建立在 TCP 之上。TCP 作为互联网最基础的传输协议，可靠性毋庸置疑，但它的设计初衷是上世纪 70 年代的网络环境，在现代高延迟、有丢包的跨境链路上，TCP 的几个固有问题开始成为瓶颈。

**队头阻塞（Head-of-Line Blocking）**是最核心的问题。TCP 保证数据有序到达，当传输中的某个数据包丢失时，即使后续的数据包已经到达接收端，TCP 也不会将它们交给上层应用——必须等到丢失的包被重传并正确接收后，整个数据流才能继续。这意味着一个数据包的丢失会阻塞同一 TCP 连接上的所有后续数据，即使这些数据属于完全不同的请求。在代理场景中，你访问 YouTube 的流量和加载 Google 搜索结果的流量可能共享同一个 TCP 连接，一个视频帧的丢包会连带阻塞搜索结果的加载。

**拥塞控制过于保守**也是一个长期存在的问题。TCP 的拥塞控制算法（Cubic、Reno 等）在检测到丢包后会大幅降低发送速率，然后缓慢恢复。这种策略在局域网或低延迟网络中没问题，但在跨境代理这种高延迟、本身就有一定丢包率的链路上，TCP 会频繁触发拥塞退避，导致实际吞吐量远低于链路的理论带宽。用户感知到的就是"带宽明明够，但速度就是上不去"。

**握手开销**同样值得关注。TCP 建立连接需要三次握手（1.5 RTT），如果加上 TLS 1.3 的握手（额外 1 RTT），一次新连接至少需要 2.5 个 RTT 的延迟才能开始传输数据。在跨太平洋链路上，单次 RTT 可能达到 150-200ms，这意味着每次新连接都有 375-500ms 的固定延迟开销。

### QUIC 的优势

QUIC 是 Google 在 2012 年开始设计、2021 年被 IETF 标准化为 RFC 9000 的传输协议。它已经是 HTTP/3 的底层传输协议，截至 2026 年，全球超过 30% 的 Web 流量通过 QUIC 传输。QUIC 针对 TCP 的几个核心缺陷提供了系统性的解决方案。

**基于 UDP 构建，消除队头阻塞**。QUIC 运行在 UDP 之上，但在用户空间实现了自己的可靠传输机制。关键区别在于：QUIC 支持多路复用的独立流（Stream）。每个流有自己的序号空间和重传逻辑，一个流中的丢包只会阻塞该流本身，不会影响同一连接中的其他流。回到前面的例子：YouTube 视频流和 Google 搜索请求分属不同的 QUIC Stream，视频帧丢包时搜索结果照常加载。

**0-RTT 或 1-RTT 连接建立**。QUIC 将传输握手和 TLS 1.3 握手合并在一起，新连接只需要 1 个 RTT 即可开始传输数据。对于之前连接过的服务器，QUIC 甚至支持 0-RTT 恢复——客户端可以在发送第一个握手包的同时附带应用数据，实现零延迟的连接建立。在高延迟的跨境链路上，这个优化的体感差异非常明显。

**内置加密**。QUIC 强制使用 TLS 1.3 加密，不仅数据负载被加密，连大部分协议头（包括包序号）也被加密。这使得中间设备（包括审查系统）更难对 QUIC 流量进行深度分析。

**更灵活的拥塞控制**。QUIC 的拥塞控制在用户空间实现，而不是像 TCP 那样在操作系统内核中。这意味着应用程序可以自由选择和定制拥塞控制算法，不受操作系统限制。Hysteria2 正是利用了这一点来实现其激进的 Brutal 拥塞控制。

**QUIC 是 HTTP/3 的基础——它就是互联网的未来**。越来越多的网站和服务已经在使用 QUIC/HTTP3，这意味着基于 QUIC 的代理流量在网络中并不异常——它只是看起来像另一个 HTTP/3 连接而已。

## Hysteria2

### 设计理念

Hysteria2 的设计目标非常明确：**在恶劣的网络条件下最大化传输吞吐量**。它不像 VLESS+Reality 那样把抗检测作为第一优先级，而是把"快"放在了首位。对于需要跨境传输大量数据的用户——看 4K 流媒体、下载大文件、视频会议——Hysteria2 解决的是一个非常实际的痛点：TCP 协议在高丢包链路上的性能塌陷。

Hysteria2 的作者在项目文档中直言不讳地表示，这个项目的核心价值就是"速度"。它的所有设计决策都围绕这个目标展开：选择 QUIC 而非 TCP、实现自定义的激进拥塞控制、简化协议开销。

### Brutal 拥塞控制

Brutal 是 Hysteria2 最具特色也最具争议的设计。理解 Brutal 需要先理解传统拥塞控制的逻辑。

标准的拥塞控制算法（Cubic、BBR 等）遵循一个基本原则：当检测到丢包或延迟增加时，降低发送速率以缓解网络拥塞。这是一种"公民意识"——我检测到网络拥堵了，那我就少发一点，给其他人让路。这种策略在正常网络中非常合理，它维持了互联网的基本秩序。

但在跨境代理场景中，丢包的原因往往不是真正的网络拥塞，而是链路本身的质量问题（海底光缆信号衰减、中间路由器过载等）。传统拥塞控制算法无法区分"拥塞性丢包"和"非拥塞性丢包"，一律执行退避策略，结果就是在这种链路上吞吐量远低于链路的实际承载能力。

Brutal 的做法截然不同：**用户直接指定目标带宽**（比如 100Mbps），Brutal 会尝试维持这个发送速率，无论网络是否出现丢包信号。它不会因为检测到丢包就降低速率。

```
# 用户配置目标带宽
bandwidth:
  up: 50 mbps
  down: 100 mbps
```

这种做法的优势是显而易见的：在丢包率 5%-10% 的跨境链路上，传统 TCP 的吞吐量可能只有理论带宽的 20%-30%，而 Brutal 可以维持接近目标带宽的发送速率（通过冗余重传来补偿丢包），实际吞吐量可以达到传统 TCP 的 3-5 倍。

但 Brutal 的劣势同样明显：

1. **对同网络其他用户"不友好"**。Brutal 不遵守传统的拥塞退避规则，会抢占其他流量的带宽。如果在共享网络中使用，可能会影响同网络其他用户的体验。
2. **流量特征异常**。正常的网络流量在丢包时会降速，但 Brutal 不会——这种"抗丢包"的行为模式与正常流量明显不同，可能被流量分析系统识别。
3. **带宽估算需要准确**。如果你设定的目标带宽远超链路实际能力，Brutal 会不断发送大量无法送达的数据包，反而加剧拥塞，导致性能更差。

Hysteria2 也支持 BBR 拥塞控制作为替代选项，适合不需要极限带宽但仍想使用 QUIC 优势的用户。

### 协议特点

**HTTP/3 伪装**。Hysteria2 的流量在网络层面看起来像标准的 HTTP/3（QUIC）连接。它使用标准的 QUIC 握手过程，对于网络中间设备而言，Hysteria2 的数据包与访问一个支持 HTTP/3 的网站产生的流量没有明显区别。服务端还可以配置一个伪装站点（masquerade），当非 Hysteria2 客户端连接时，返回真实网页内容。

**简化的认证机制**。与 VLESS/VMess 使用 UUID 不同，Hysteria2 使用简单的密码认证。这降低了配置复杂度，但在多用户管理场景下不如 UUID 方便。

**内置 ACME 支持**。Hysteria2 可以自动通过 Let's Encrypt 申请和续期 TLS 证书，不需要额外的证书管理工具。当然，这需要你拥有一个域名。

**端口跳跃（Port Hopping）**。这是 Hysteria2 的一个重要抗封锁特性。传统代理固定监听一个端口，一旦该端口被封锁，节点就失效了。Hysteria2 支持在多个 UDP 端口之间轮换连接——当某个端口被封时，客户端自动切换到下一个端口。这类似于"打游击"的策略，大幅增加了封锁方的成本。

```yaml
# 端口跳跃配置示例
# 客户端可以指定端口范围
server: example.com:20000-40000
```

**同时支持代理和 VPN 模式**。Hysteria2 既可以作为 SOCKS5/HTTP 代理使用，也可以配置 TUN 模式实现全局流量代理。

### 部署示例

Hysteria2 的部署非常简洁，一个典型的服务端配置如下：

```yaml
listen: :443

tls:
  cert: /path/to/cert.pem
  key: /path/to/key.pem

auth:
  type: password
  password: your-password

masquerade:
  type: proxy
  proxy:
    url: https://example.com
    rewriteHost: true
```

这个配置做了以下几件事：监听 443 端口（UDP），使用指定的 TLS 证书，通过密码认证用户身份，对非认证请求反向代理到 example.com（伪装成正常网站）。

如果使用内置 ACME 自动申请证书：

```yaml
listen: :443

acme:
  domains:
    - your-domain.com
  email: your-email@example.com

auth:
  type: password
  password: your-password

masquerade:
  type: proxy
  proxy:
    url: https://example.com
    rewriteHost: true
```

整个部署只需要一个二进制文件加一个配置文件，相比需要配置 Nginx + Xray 的方案要简单很多。

### 优势总结

- 在高丢包网络上的吞吐量表现极其出色，远超 TCP 方案
- 部署极其简单——单二进制、单配置文件
- HTTP/3 伪装，流量外观与正常 QUIC 流量相似
- 端口跳跃增加了抗封锁的韧性
- 社区活跃，开发迭代快速，客户端支持广泛（sing-box、Clash Meta/mihomo、Shadowrocket 等）
- 内置 ACME 证书自动管理

### 劣势总结

- 基于 UDP，部分运营商会限速或封锁 UDP 流量
- 中国部分 ISP（尤其移动网络）对 UDP 有激进的 QoS 策略
- Brutal 模式产生的流量模式与正常网络行为不符，存在被检测的风险
- 相比 TCP 方案（如 VLESS+Reality），协议成熟度和抗检测能力仍有差距
- 没有类似 Reality 的主动探测防御机制——虽然有 masquerade，但其伪装深度不及 Reality

## TUIC

### 设计理念

TUIC（读作"too-ick"）同样是基于 QUIC 的代理协议，但它的设计哲学与 Hysteria2 截然不同。如果说 Hysteria2 的关键词是"极限带宽"，那 TUIC 的关键词是"规范的 QUIC 代理"。

TUIC 的目标不是突破网络极限，而是在 QUIC 的框架内实现一个标准的、行为规范的代理协议。它不使用激进的拥塞控制，不试图对抗网络的丢包信号，而是利用 QUIC 本身的多路复用、低延迟握手等特性来提升代理性能。

### 协议特点

**多路复用 QUIC Stream**。TUIC 充分利用了 QUIC 的多流特性，每个代理请求使用独立的 QUIC Stream，真正实现了无队头阻塞的并发传输。这是对 TCP 方案最直接的改进——即使某个请求遇到丢包，其他请求的传输不受影响。

**UUID 认证**。与 VLESS/VMess 生态保持一致，TUIC 使用 UUID + Password 的认证方式。这使得它更容易融入现有的代理管理体系，对于已经熟悉 V2Ray/Xray 生态的用户来说几乎没有学习成本。

**0-RTT 支持**。TUIC 支持 QUIC 的 0-RTT 连接恢复，对于曾经连接过的服务器，客户端可以在第一个握手包中就附带代理请求，进一步降低连接延迟。

**标准拥塞控制**。TUIC 使用 BBR 或 Cubic 等标准的拥塞控制算法，不会像 Brutal 那样激进地抢占带宽。这意味着 TUIC 的流量行为模式更接近正常的网络流量，被流量分析系统识别的风险相对较低。

**QUIC UDP Relay**。TUIC 支持通过 QUIC 连接转发 UDP 数据，对于需要 UDP 代理的应用（如游戏、VoIP）提供了原生支持。这是 TUIC 相对于某些 TCP 方案的一个实际优势。

### 与 Hysteria2 的区别

| 维度 | Hysteria2 | TUIC |
|------|-----------|------|
| 拥塞控制 | Brutal（激进）/ BBR | BBR / Cubic（标准） |
| 认证方式 | 密码 | UUID + Password |
| 伪装能力 | HTTP/3 伪装 + masquerade | 标准 QUIC |
| 端口跳跃 | 支持 | 不支持 |
| 客户端支持 | 广泛（sing-box、mihomo、Shadowrocket 等） | sing-box 为主 |
| 开发活跃度 | 活跃 | 较低（项目维护减少） |
| 流量特征 | 较明显（Brutal 模式下） | 较标准 |
| 适用场景 | 高带宽需求、对速度要求极高 | 通用代理、对行为规范性有要求 |
| 极端吞吐 | 极高（Brutal 模式） | 标准（受拥塞控制约束） |
| 抗封锁韧性 | 较强（端口跳跃） | 一般 |

简单概括：Hysteria2 是"速度优先、规则次之"的激进派，TUIC 是"按规矩来、但用更好的底层"的温和派。

### TUIC 的现状

需要诚实地说明：截至 2026 年中，TUIC 的项目维护状态不太理想。原作者的活跃度明显下降，代码仓库的更新频率大幅降低。虽然 sing-box 仍然支持 TUIC v5 协议，但生态的整体发展已经明显不如 Hysteria2。

这不意味着 TUIC 不能用——它在技术上仍然是一个合理的协议选择，尤其是你已经在使用 sing-box 且需要一个行为规范的 QUIC 代理时。但如果你是新用户在 QUIC 协议之间选择，Hysteria2 无疑是更务实的选项。

## QUIC 协议在中国的现状

### UDP 封锁与 QoS 问题

QUIC 协议在中国大陆面临的最大障碍不是协议本身的安全性问题，而是 UDP 传输层的限制。这是一个很多用户在选择 QUIC 方案前容易忽略的关键因素。

中国各大运营商对 UDP 流量的态度存在明显差异。总体而言，中国移动对 UDP 的限制最为严格，许多地区的移动网络会对非标准 UDP 流量（即不是 DNS、NTP 等常见 UDP 服务的流量）进行激进的 QoS 限速，严重时甚至直接丢弃。中国电信和中国联通的情况相对好一些，但也因地区不同而有显著差异。

移动网络（4G/5G）通常比宽带更严格地限制 UDP。这是因为移动基站的带宽资源更稀缺，运营商倾向于优先保障 TCP 流量。如果你主要在手机上使用代理，QUIC 方案可能不是最佳选择。

具体到 QUIC 443 端口的情况：由于 QUIC/HTTP3 已经被主流网站广泛使用（Google、YouTube、Facebook 等），完全封锁 UDP 443 端口的成本对运营商来说比较高——这会影响正常的 Web 浏览体验。因此大部分运营商不会完全封锁该端口，但可能会对其进行限速。

### 检测风险

GFW 对 QUIC 流量的分析能力正在持续增强。虽然目前尚未实现对 Hysteria2/TUIC 的精准识别（不像裸 VMess 那样可以被准确封杀），但以下几个方面的检测风险需要关注：

**Brutal 模式的流量特征**。正如前文所述，Brutal 拥塞控制产生的流量模式（在丢包时不降速）与正常网络行为明显不同。流量分析系统可以通过统计包速率与丢包率的相关性来识别 Brutal 流量。

**单一 IP 的 QUIC 连接集中度**。正常情况下，QUIC 流量主要指向 Google、Cloudflare 等大型 CDN 的 IP 地址。如果 GFW 观察到大量 QUIC 连接指向一个非主流 CDN 的单一 IP，这本身就是一个异常信号。

**连接时长和流量模式**。正常的 HTTP/3 连接通常是短暂的（加载网页资源），而代理连接可能持续数小时且保持稳定的高流量。这种行为模式与标准的 Web 浏览有明显差异。

**检测风险评估**：中等，且呈上升趋势。目前 QUIC 方案的安全性低于 VLESS+Reality 这样的 TCP 方案，但高于裸 VMess 或未加密的 Shadowsocks。

## 该不该用 QUIC 协议

### 适合使用的场景

- **ISP 对 UDP 限制不严**：你的宽带运营商不会对 UDP 流量进行显著的 QoS 限速。可以通过连接一个 Hysteria2 节点进行速度测试来验证。
- **需要高带宽传输**：4K/8K 流媒体、大文件下载、视频会议等场景，且你的 TCP 节点在高峰时段性能下降明显。
- **TCP 节点高峰拥堵**：晚间高峰时段 TCP 节点速度大幅下降，Hysteria2 的 Brutal 拥塞控制可能突破 TCP 的性能瓶颈。
- **作为备用协议**：即使 TCP 方案是你的主力，也应该准备一个 QUIC 方案作为后备。在 TCP 被针对性封锁时，QUIC 方案可能仍然可用（反之亦然）。

### 不适合的场景

- **ISP 封锁或严格限速 UDP**：如果你的网络环境对 UDP 有明确的限制，QUIC 方案的性能可能反而不如 TCP 方案。
- **主要使用移动网络**：4G/5G 网络对 UDP 的限制通常更严格。
- **隐蔽性是首要需求**：如果你在高审查环境中且首要目标是"不被发现"，VLESS+Reality 基于 TCP 的方案在抗检测方面仍然更成熟、更可靠。
- **游戏加速**：虽然 QUIC 理论上适合实时应用，但如果运营商对 UDP 有 QoS 限制，游戏体验反而可能更差。且游戏对延迟波动非常敏感，Brutal 模式的激进发包策略可能导致延迟不稳定。

### 推荐策略

对于大多数用户，最务实的做法是**多协议组合**：

- **主力协议**：VLESS + Reality（TCP，最可靠，抗检测能力最强）
- **备用协议**：Hysteria2（UDP 友好环境下的高带宽选项）
- **客户端配置**：利用 sing-box 或 mihomo 的自动切换功能，当 TCP 节点延迟过高时自动切换到 Hysteria2 节点

不要把所有节点都放在同一种协议上。TCP 和 UDP 属于不同的传输层，运营商的封锁策略通常也是分开的——这意味着 TCP 被封时 UDP 可能还能用，反之亦然。保持协议多样性是最基本的风险分散策略。

## 常见问题

### Q: Hysteria2 和 VLESS+Reality 只能选一个吗？

不用。两种协议完全可以同时部署在同一台服务器上——VLESS+Reality 监听 TCP 443，Hysteria2 监听 UDP 443，互不干扰。很多机场也同时提供两种协议的节点。你可以在客户端设置自动切换策略：TCP 节点延迟高于阈值时自动切换到 Hysteria2，反之亦然。这种混合策略在实践中非常有效。

### Q: 怎么测试我的网络支不支持 UDP/QUIC？

最直接的方法：连接一个 Hysteria2 节点，尝试正常使用。如果能连上且速度正常，说明你的网络对 UDP 没有严格限制。如果完全连不上但 TCP 节点正常工作，基本可以确认 UDP 被限制了。

你也可以做一个更细致的测试：同时测试 Hysteria2 节点和 TCP 节点的速度。如果 Hysteria2 的速度明显低于 TCP 节点（比如只有后者的一半甚至更低），说明你的网络虽然没有完全封锁 UDP，但对 UDP 进行了 QoS 限速。在这种情况下，QUIC 方案的性价比不高。

另一个简单的测试方式是检查 YouTube 是否通过 QUIC 协议加载。在 Chrome 浏览器中打开 YouTube，按 F12 打开开发者工具，切换到 Network 面板，查看请求的 Protocol 列是否显示"h3"（即 HTTP/3/QUIC）。如果 YouTube 可以正常使用 h3 协议，说明你的网络对 QUIC 比较友好。

### Q: TUIC 还值得用吗？

坦率地说，如果你是新用户在 QUIC 方案中做选择，推荐直接选 Hysteria2。原因有几个：Hysteria2 的社区明显更活跃，客户端支持更广泛（Clash Meta/mihomo、sing-box、Shadowrocket、Surge 等主流客户端都已支持），文档和教程资源更丰富。TUIC 原作者的活跃度明显下降，项目的未来发展存在不确定性。

如果你已经在使用 TUIC 且运行稳定，没必要急着迁移——它在技术上仍然是一个可靠的协议。但如果遇到问题需要排障，你可能会发现 TUIC 相关的社区支持和资源已经明显少于 Hysteria2。

### Q: 端口跳跃是什么意思？

端口跳跃（Port Hopping）是 Hysteria2 的一项抗封锁功能。传统代理固定在一个端口上运行（比如 UDP 443），一旦这个端口被运营商封锁，节点就失效了。端口跳跃允许 Hysteria2 客户端在一个预设的端口范围内（比如 20000-40000）轮换连接端口——封了 20001 就用 20002，封了 20002 就用 20003。服务端通过 iptables/nftables 将这个端口范围的流量全部转发到 Hysteria2 实际监听的端口上。

这种策略大幅增加了封锁方的成本：它需要同时封锁整个端口范围才能完全阻断连接，而不是只封一个端口。这也是 Hysteria2 相对于 TUIC 的一个实际优势——TUIC 不支持端口跳跃。

### Q: Brutal 模式的带宽应该设多少？

一个常见的误区是把 Brutal 的目标带宽设得越高越好。实际上，这个值应该设为你的实际链路带宽的 80%-90%，而不是一个脱离实际的高值。如果你的宽带是 100Mbps，设 80-90Mbps 是合理的；设 500Mbps 不会让你更快，反而会导致大量无效重传，加剧网络拥塞。

如果不确定该设多少，建议先用 BBR 模式（不使用 Brutal）进行速度测试，得到一个基准带宽值，然后在此基础上设置 Brutal 的目标带宽。如果 BBR 模式下已经能达到满意的速度，不用 Brutal 也完全可以。
