# GFW 工作原理全解：从 DPI 到主动探测

> **摘要**：GFW（Great Firewall）并非单一技术，而是多种检测手段的组合系统。本文拆解其核心检测机制——DNS 污染、深度包检测（DPI）、TLS 指纹识别、主动探测、流量行为分析——并讨论各机制对应的规避策略，帮助你理解封锁与反封锁的技术博弈。

## GFW 不是一堵"墙"

很多人把 GFW 想象成一个简单的黑名单过滤器，实际上它是一个多层次、持续演进的流量分析系统。理解这一点很重要：**没有一种协议能"永远安全"，因为检测侧也在持续迭代。**

## 检测机制逐层拆解

### 第一层：DNS 污染

**原理**：
<!-- 
- GFW 监听 UDP 53 端口的 DNS 查询
- 对命中黑名单的域名抢先返回错误的 IP 地址
- 这是最古老也最容易绕过的封锁方式
-->

**绕过方式**：
<!-- DoH/DoT、Fake-IP 模式、远端 DNS 解析 -->

**为什么它仍然有效**：
<!-- 大量普通用户不会配置 DNS，默认走运营商 DNS 就被污染了 -->

### 第二层：IP 封锁

**原理**：
<!-- 维护一个被封锁的 IP 段列表，TCP 连接阶段直接 RST 或丢包 -->

**特点**：
<!-- 粗暴但有效，常用于封锁已知的 VPN 服务器 IP 段 -->

**应对思路**：
<!-- 节点 IP 轮换、使用 CDN 中转、Anycast IP -->

---

### 第三层：深度包检测（DPI）

**原理**：
<!-- 
- 不仅看 IP 和端口，而是分析数据包的内容和格式
- 识别特定协议的指纹特征
- 已知能识别的协议：未加密的 SS（旧版）、裸 VMess、OpenVPN、WireGuard 等
-->

**它能看到什么**：
<!-- 
- TLS 握手的明文部分（Client Hello、Server Hello）
- 数据包大小分布和时序特征
- 协议握手的固定模式
-->

**它看不到什么**：
<!-- 
- TLS 加密后的应用层数据
- 正确实现的加密协议内容
-->

---

### 第四层：TLS 指纹识别（JA3/JA4）

**这是当前最关键的检测维度之一。**

**原理**：
<!-- 
- TLS Client Hello 包含大量明文元数据：支持的加密套件、扩展列表、椭圆曲线参数等
- 这些参数的组合构成一个"指纹"，不同的 TLS 库（Go、Rust、uTLS、浏览器原生）会产生不同的指纹
- JA3/JA4 算法将这些参数哈希为一个指纹值
- GFW 可以通过比对指纹判断连接是否来自真实浏览器
-->

**为什么这对代理很致命**：
<!-- 
- 大多数代理工具用 Go 的 crypto/tls 库，其指纹与 Chrome/Firefox 明显不同
- 即使用了 uTLS 模拟浏览器指纹，模拟的完整度和更新及时性是个问题
- Reality 协议通过直接转发目标站点的 Server Hello 来解决这个问题
-->

**当前状态**：
<!-- 基于你的研究，说明 GFW 在 TLS 指纹识别上的实际执行力度 -->

---

### 第五层：主动探测

**原理**：
<!-- 
- GFW 不只是被动监听，还会主动连接可疑服务器进行验证
- 探测行为：模拟客户端发送握手请求，观察服务端的响应是否符合代理协议特征
- 对 Shadowsocks 和早期 V2Ray 非常有效
-->

**探测策略**：
<!-- 
- 重放攻击：抓取并重放之前的握手包
- 随机数据探测：发送随机数据观察服务端是否返回特定错误
- 协议探测：尝试以不同协议与服务端握手
-->

**防御方式**：
<!-- 
- Reality 的设计就是为了对抗主动探测：探测者看到的是真实的目标网站
- SS-2022 加入了重放保护
- 好的服务端实现在收到无效请求时的行为与正常 Web 服务器一致
-->

---

### 第六层：流量行为分析（统计分析）

**原理**：
<!-- 
- 即使协议本身无法被 DPI 识别，长期的流量模式也可能暴露
- 分析维度：连接频率、数据包大小分布、连接持续时间、时段规律
- 一个 IP 长期与海外单一 IP 保持高频加密连接，本身就是异常信号
-->

**为什么这很难对抗**：
<!-- 
- 这是统计层面的检测，不依赖协议指纹
- 你可以完美伪装协议，但使用模式本身会暴露你
- 目前没有完美解决方案，只能通过降低异常度来降低风险
-->

**缓解策略**：
<!-- 
- 多节点轮换，避免长期连接单一 IP
- 使用中转节点分散源 IP
- 混入正常流量
-->
